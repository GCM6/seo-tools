## 第一部分：SEO 诊断——行业前沿 vs 方案覆盖

### 1.1 诊断方法论：从"清单审计"到"漏斗约束分析"

2026 年专业 SEO 诊断已经从"逐项检查清单"升级为**先定位约束瓶颈，再针对性深挖**的分层模型：

```
[行业前沿] 漏斗约束诊断（Funnel-Constraint Analysis）
┌─────────────────────────────────────────────────────┐
│ 第一步：识别约束类型                                    │
│   ├── 系统性故障？（抓取/索引/渲染完全断裂）              │
│   ├── 可见性问题？（有索引但无展示/排名）                 │
│   └── 转化问题？  （有展示/流量但无转化）                 │
│                                                       │
│ 第二步：按约束类型选择诊断路径                           │
│   系统性 → 技术层深挖                                   │
│   可见性 → 关键词/内容/权威层                            │
│   转化   → 意图匹配/页面体验/CTA                        │
│                                                       │
│ 第三步：优先修最大约束，而非逐项修所有问题                 │
└─────────────────────────────────────────────────────┘
```

> [!IMPORTANT]
> **方案当前状态**：方案采用的是"五支柱全面扫描"模式——所有规则都跑一遍，用 Impact × Effort 排优先级。这本身**没问题**，但缺少一个**诊断入口的漏斗分诊层**——告诉用户"你的核心瓶颈在哪"，而不是一上来就给 40+ 条 finding。
>
> **差什么**：报告需要在执行摘要中增加一个**"约束定位"模块**——基于证据自动判断站点当前卡在漏斗哪一层，指导用户先看哪个支柱。

### 1.2 四层技术诊断（传统抓取 → AI Agent 可达）

2026 年的技术诊断已从单一的"Googlebot 能不能抓"扩展为四层：

| 层 | 诊断什么 | 方案覆盖 | 缺口 |
|---|---|---|---|
| **L1 传统抓取** | robots.txt / sitemap / 状态码 / 重定向链 | ✅ T01-T08 完整覆盖 | — |
| **L2 AI Agent 可达** | AI 爬虫（搜索型/训练型）的 robots 规则 + CDN/WAF 误封 | ✅ G01/G02 覆盖 | — |
| **L3 JS 渲染** | 初始 HTML vs 渲染后内容差异、AI 爬虫不执行 JS | ✅ T10/G03 覆盖 | — |
| **L4 日志层** | 服务器日志分析：各类爬虫的实际抓取行为 vs 预期 | ❌ **完全缺失** | 见下 |

> [!WARNING]
> **日志分析（Log File Analysis）是行业公认的"诊断真相来源"**——它是唯一能直接回答"Googlebot 实际花了多少预算在哪些页面"、"AI 爬虫到底来了没有"的数据。但这需要用户提供服务器日志，实现复杂度高。
>
> **建议**：Phase E 或 V2 中作为可选高级功能评估，当前在 §10"明确不做"中声明。

### 1.3 语义/实体维度——SEO 的新基础层

2026 年 SEO 已从"关键词优化"全面转向"实体优化"：

```
[行业前沿] 实体 SEO 诊断维度

1. 品牌实体识别
   ├── Google 知识图谱是否有品牌 Panel？
   ├── sameAs 属性是否连接权威节点（Wikidata / LinkedIn / Crunchbase）？
   └── NAP（名称/地址/电话）跨平台一致性？

2. 主题权威度（Topical Authority）
   ├── 内容是否形成了 Pillar-Cluster 结构？
   ├── 主题覆盖的完整度（vs SERP Top-5 竞品的子话题覆盖）？
   ├── 子页面是否自然获得了外链（不仅仅是首页）？
   └── 用户搜索"品牌+主题"的频次（GSC 品牌组合词）？

3. 内容语义深度
   ├── NLP 语义覆盖 vs 竞品 Top-5 的 TF-IDF / 实体覆盖差异
   ├── Information Gain：是否提供了独有数据/原创观点？
   └── 答案前置：前 2-3 段是否包含可独立成答的简洁回答？
```

> **方案当前状态**：
> - ✅ C07 覆盖了 GEO 内容特征（统计数据/引述/来源引用）
> - ✅ C08 覆盖了答案前置检测
> - ⚠️ C05 覆盖了 Schema 存在性和校验，但**没有覆盖实体 SEO 维度**（sameAs / 知识图谱 / NAP 一致性）
> - ❌ **主题权威度诊断完全缺失**——没有 Pillar-Cluster 结构分析、没有子话题覆盖度检测
> - ❌ **NLP 语义覆盖对比缺失**——没有与竞品 Top-5 的内容深度对比

### 1.4 内容生命周期管理（Content Decay + ROT 分析）

行业前沿已将内容视为有**生命周期**的资产：

```
[行业前沿] 内容生命周期诊断

Content Decay（内容衰退）
  触发条件：页面级 GSC 点击/展示 6-12 月持续下降
  输出三级建议：
    ├── Refresh（刷新）：高价值、小幅下滑、有外链 → 更新数据和案例
    ├── Consolidate（合并）：多页蚕食同意图 → 合并 + 301
    └── Prune（剪枝）：零流量、无外链、无价值 → 下线或 noindex

ROT 分析（Redundant / Outdated / Trivial）
  ├── Redundant：同站内多页覆盖同一话题（蚕食）
  ├── Outdated：发布日期超过阈值 + GSC 下降 + 竞品新内容出现
  └── Trivial：字数低 + 零展示 + 无外链 + 无内链支撑
```

> **方案当前状态**：
> - ⚠️ K06 覆盖了关键词蚕食（Redundant 的一部分）
> - ⚠️ K08 覆盖了断崖式下跌（Content Decay 的极端情况）
> - ❌ **渐进式衰退检测缺失**（6-12 月滚动趋势分析）
> - ❌ **ROT 分析框架缺失**——没有系统性的"该刷新/该合并/该下线"三级输出

## 第二部分：GEO 诊断——行业前沿 vs 方案覆盖

### 2.1 GEO 诊断的四支柱框架

2026 年的 GEO 诊断已形成相对成熟的方法论：

```
[行业前沿] AI 可见性审计四支柱

支柱 A：可达性（Access & Crawlability）
  ├── AI 搜索爬虫是否被屏蔽？                      → 方案 G01 ✅
  ├── CDN/WAF 是否误封？                          → 方案 G02 ✅
  ├── JS 渲染内容对 AI 不可见？                     → 方案 G03/T10 ✅
  └── Bing 是否收录？（ChatGPT 依赖 Bing）          → 方案 G04 ✅

支柱 B：实体与信任权威（Entity & Trust Authority）
  ├── 品牌在知识图谱中是否被识别？                   → ❌ 缺失
  ├── Schema sameAs 是否连接权威节点？              → ❌ 缺失
  ├── 第三方平台的品牌一致性？                      → ⚠️ G07 部分覆盖
  └── 品牌搜索量 vs AI 引用相关性？                 → ❌ 缺失

支柱 C：内容可引用性（Content Citability）
  ├── 内容是否有统计/引述/来源引用？                 → 方案 C07 ✅
  ├── 答案前置（BLUF 格式）？                       → 方案 C08 ✅
  ├── 结构化可扫描（清晰标题层级、列表、表格）？       → ⚠️ 隐含在轻检中
  └── 独有数据/原创观点（Information Gain）？         → ❌ 缺失

支柱 D：可见性度量（Share of Model / Citation）
  ├── 分引擎探针（ChatGPT/Perplexity/Gemini）       → 方案已有探针 ✅
  ├── 品牌出现率 + Wilson 区间                      → 方案 §4.1 ✅
  ├── 竞品对比 SoV                                 → 方案 Q02/G05/G06 ✅
  └── 引用情感分析（正面/负面/中性）                  → ❌ 缺失
```

### 2.2 AI 引用管线机制——对诊断设计的启示

最新研究揭示了 AI 模型选择引用来源的管线机制：

```
用户查询 → 意图解析 → 混合检索（语义向量 + BM25 关键词）
         → 交叉编码器 Reranker（精选 Top 5-10 段落）
         → LLM 生成 + Grounding（每个声称绑定来源段落）
         → 输出答案 + 引用
```

**关键发现**（对方案设计的启示）：

| AI 引用偏好因素 | 方案覆盖 | 差距 |
|---|---|---|
| **结构清晰度**：40-60 词段落、列表、表格易被引用 | ⚠️ C08 部分覆盖（答案前置） | 缺少"结构可扫描性"的系统检查 |
| **多源共识**：多个可信来源说同一件事 → 更容易被引用 | ✅ G07 第三方语料检测 | — |
| **新鲜度**：datePublished / 更新时间戳 | ✅ C06 中的 datePublished | — |
| **品牌搜索量**：品牌知名度 → AI 信任度（Spearman 0.664） | ❌ **方案未检测品牌搜索量** | 可用 GSC 品牌词展示量 |
| **引擎编辑身份差异**：不同 AI 引擎偏好不同来源平台 | ✅ 方案已分引擎报告 | — |
| **排名解耦**：Page 3 的内容也可能被 AI 引用 | ✅ 方案已意识到（§2 调研） | — |

### 2.3 方案 GEO 诊断的评估结论

> [!TIP]
> **方案的 GEO 诊断在支柱 A（可达性）和支柱 D（可见性度量）上已经做到了行业领先**——尤其是分引擎探针 + 分引擎报告 + Wilson 区间的统计严谨性，远超 Ahrefs/Semrush。
>
> **主要差距在支柱 B（实体与信任）和支柱 C（内容可引用性）**——这两个维度恰好是 AI 引用管线中"为什么选你而不选别人"的关键判断依据。

## 第三部分：SEO 优化建议生成——行业前沿 vs 方案覆盖

### 3.1 从"发现问题"到"给出可执行建议"

2026 年行业对"优化建议生成"的标准已从"告诉你哪里有问题"升级为"告诉你具体怎么改"：

```
[行业前沿] 优化建议的三个层次

Level 1: What（发现了什么问题）           ← 所有工具都能做
Level 2: Why + Expected Impact            ← 专业工具的标准
Level 3: How（具体 Action Item + 模板）   ← 领先工具的差异化
```

| 建议层次 | 行业前沿 | 方案覆盖 |
|---|---|---|
| **L1 发现** | finding 带证据 | ✅ 规则引擎 + evidenceRefs |
| **L2 原因+影响** | what/why/expected_impact/effort | ✅ 建议模板已声明 |
| **L3 具体执行** | | |
| → 技术修复 | 给出具体代码片段 / 配置建议 | ⚠️ 有建议模板但**未见具体代码/配置示例** |
| → 内容优化 | Content Brief（实体地图/标题骨架/引用要求） | ⚠️ 有 prompt assembler 但**未设计 Brief 格式** |
| → 外链/权威 | 具体的目标列表 + 策略模板 | ⚠️ 有经验级策略但**粒度不够** |

> **差什么**：方案的建议系统架构完整（finding → recommendation → prompt），但**建议模板的具体内容深度**还需要补充。尤其：
> - 技术类建议需要有**代码示例/配置片段**（如 robots.txt 修复示例、canonical 修复模板）
> - 内容类建议需要生成**结构化 Content Brief**（不仅仅是一段话的建议）

### 3.2 Content Brief 的前沿结构

2026 年行业领先的 Content Brief 已远超"给几个关键词"：

```
[行业前沿] Content Brief 模板

1. 目标关键词 + 意图分类
2. SERP 分析摘要（Top-5 的内容类型、字数、结构）
3. 实体地图（必须覆盖的核心实体和关系）
4. 标题骨架（H1 → H2 → H3 建议结构）
5. E-E-A-T 质量门控
   ├── 必须有作者署名
   ├── 必须有 ≥3 处外部引用来源
   ├── 必须有 datePublished
   └── 推荐包含第一手经验证据
6. GEO 格式要求
   ├── 答案前置（BLUF：在每个 H2 下的前 40-60 词给出直接答案）
   ├── 至少一处数据引述
   └── 结构化可扫描（列表/表格/对比）
7. AI 初稿必须人工终审声明
```

> **方案当前状态**：方案的 prompt assembler 主要生成"给 AI 用的内容生成 prompt"，但**没有设计面向人类作者/编辑的 Content Brief 格式**。这是一个重要的交付物缺口——因为很多用户的内容团队不用 AI 生成全文，而是需要一份**指导人类写作的 Brief**。

## 第五部分：需要新增设计的具体内容

基于以上对照，以下是方案距离"行业前沿专业诊断与优化工具"还需补充的设计点，按优先级排列：

### 🔴 高优先级（直接影响诊断专业度）

#### 1. 报告入口增加"约束定位"模块

**为什么需要**：40+ 条 finding 出来后用户不知道从哪看起。

**怎么做**：在执行摘要中增加一个自动判断模块——
```
约束定位算法：
  IF P1 有 error 级 finding（抓取/索引/渲染断裂）
    → 主约束 = "系统性基础问题"，引导先看 P1
  ELSE IF P3 关键词现状为空/极稀疏（GSC 未接或零数据）
    → 主约束 = "可见性数据不足"，引导接 GSC 或配 DataForSEO
  ELSE IF P3 有大量 K03/K04 缺口词 + P5 外链/语料弱
    → 主约束 = "权威与内容竞争力不足"，引导看 P3+P5
  ELSE
    → 主约束 = "精细优化阶段"，按 Impact 排序
```

**属于哪个 Phase**：Phase E（综合报告），实现成本低。

---

#### 2. 实体 SEO / 知识图谱诊断

**为什么需要**：2026 年 SEO 和 GEO 的共同基础是"品牌实体被机器识别和信任"。

**怎么做**：新增检查项（可放 P5 或单独一组）——
```
E01  Schema Organization 是否含 sameAs 指向权威节点
     （Wikidata / LinkedIn / Crunchbase）
     严重度 notice | claim measured_hard
     来源：schema-extractor 现有能力扩展

E02  Google Knowledge Panel 存在性
     （DataForSEO SERP 品牌词查询，检查 knowledge_graph 字段）
     严重度 notice | claim measured_sample
     来源：Phase C DataForSEO

E03  品牌 NAP 跨平台一致性（进阶，可选）
     难以自动化，建议仅在报告中以检查清单提示
```

**属于哪个 Phase**：E01 可放 Phase A（扩展 schema-extractor），E02 放 Phase C。

---

#### 3. 主题权威度诊断

**为什么需要**：Google 2026 年重点奖励"在一个领域有深度覆盖"的站点，而非"什么都写一点"。

**怎么做**：利用已有的全站爬取 + URL 模板聚类——
```
TA01 内容聚类覆盖度
     全站页面 → URL 模板聚类（已有）→ 按模板/目录识别"话题群"
     → 每个话题群的页面数、内链密度、GSC 聚合展示量
     → 找出"有话题但覆盖浅"或"话题间无内链连接"的问题

TA02 Pillar-Cluster 结构完整性
     检测是否存在 Hub 页面（inboundLinkCount 高 + 同话题群内
     多个子页面指向它）
     → 没有 Hub 页的话题群 = "缺乏主题权威结构"
     严重度 notice | claim inferred
```

**属于哪个 Phase**：Phase B 或 C（依赖 GSC 数据交叉验证更佳）。

---

#### 4. 内容衰退 + ROT 生命周期诊断

**为什么需要**：2026 年内容不是写完就完了，需要持续管理。

**怎么做**：
```
K09  内容衰退检测
     GSC 页面级 点击/展示 近 6 月 vs 前 6 月
     下降 >30% 且当前展示量 > 阈值 → 标为"衰退中"
     severity warning | claim inferred
     → 建议模板：Refresh / Consolidate / Prune 三级

K10  ROT 分析（Redundant / Outdated / Trivial）
     R：contentHash 重复 或 K06 蚕食命中 → 合并
     O：datePublished 超 18 月 + GSC 下降 → 刷新
     T：字数 < 300 + GSC 零展示 + 零外链 + 零内链 → 下线
     severity notice | claim inferred
```

**属于哪个 Phase**：Phase B（依赖 GSC 数据）。

---

#### 5. 内容可扫描性/结构化检测

**为什么需要**：AI 引用管线偏好"结构清晰、可快速提取"的内容。

**怎么做**：扩展轻检 `fetchLightCheck`——
```
新增抽取字段：
  listCount        ← <ul>/<ol> 数量
  tableCount       ← <table> 数量
  avgParagraphLen  ← 段落平均长度（过长 = 不易扫描）
  h2AsQuestionRate ← H2 中以疑问词开头的比例

C11  内容结构可扫描性不足
     重点页/商业页：无列表 + 无表格 + 平均段落 >150 词
     → 建议"拆分为列表/表格/Q&A 格式，提升 AI 可引用性"
     severity notice | claim inferred
```

**属于哪个 Phase**：Phase A 或 D（轻检字段扩展，低成本）。

---

### 🟡 中优先级（增强优化建议质量）

#### 6. Content Brief 作为优化交付物

**为什么需要**：用户拿到"这个页面需要优化内容"的建议后，下一步是"怎么写"。

**怎么做**：在 prompt assembler 之外，新增 Brief Generator——
```
输入：finding（如 K03 缺口词 + Q03 竞品内容形态）
输出：结构化 Content Brief
  ├── 目标关键词 + 意图
  ├── SERP Top-5 内容类型/字数/结构摘要（来自竞品轻检）
  ├── 推荐标题骨架（H1/H2/H3）
  ├── 必须覆盖的实体/子话题清单
  ├── E-E-A-T 要求（署名/引用/日期）
  ├── GEO 格式要求（BLUF/数据引述/列表化）
  └── "AI 初稿须人工终审"声明
```

**属于哪个 Phase**：Phase D 或 E。

---

#### 7. 技术类建议增加代码/配置示例

**为什么需要**：用户看到"robots.txt 屏蔽了重点页"后，需要知道"具体改哪行"。

**怎么做**：在建议模板中为技术类 finding 增加 `fixSnippet` 字段——
```
T01 → fixSnippet: "移除 robots.txt 中的 Disallow: /[path]"
T04 → fixSnippet: "<link rel='canonical' href='https://...'>"
G01 → fixSnippet: "在 robots.txt 中添加：\nUser-agent: OAI-SearchBot\nAllow: /"
T09 → fixSnippet: "考虑对 [resource] 使用 lazy loading 或预加载"
```

**属于哪个 Phase**：Phase A（建议模板丰富化）。

---

#### 8. 品牌搜索量度量（GEO 信任代理指标）

**为什么需要**：品牌搜索量与 AI 引用相关性 0.664（方案已引用此数据），但没有实际度量。

**怎么做**：
```
利用 GSC 数据（Phase B）：
  品牌词展示量 = 筛选包含品牌名的 query 的总展示量
  品牌词 CTR = 品牌词的平均 CTR

利用 DataForSEO（Phase C）：
  品牌搜索量 = DataForSEO Labs 查询品牌词的月均搜索量
  竞品品牌搜索量 = 对比

→ 在报告中展示"品牌搜索量 vs 竞品"对比，
  作为 P5 权威维度的一个子指标
```

**属于哪个 Phase**：Phase B/C 自然落地。

---

#### 9. 引用情感方向检测

**为什么需要**：被 AI 引用≠好事——如果 AI 说"X 品牌评价较差"，那是负面引用。

**怎么做**：在现有探针的 SoV 聚合中增加——
```
现有探针返回 AI 响应文本（已有）
→ 新增后处理：对包含品牌名的句子做简单情感分类
   （positive / neutral / negative / comparison）
→ 在 G05 报告中除了出现率，增加"情感分布"
→ 如果负面引用占比高，触发一条建议
   severity warning | claim inferred
```

**属于哪个 Phase**：Phase D（GEO 深化）。

---

### 🟢 低优先级（边界声明或长期规划）

#### 10. 日志分析集成

服务器日志是"真相来源"，但用户提供日志的门槛高。建议在 §10 中声明"不做日志分析集成（V2 评估）"。

#### 11. NLP 语义覆盖对比

需要调用 NLP API 或 LLM 对比本站 vs 竞品的语义覆盖，实现复杂度高，claim 上限 inferred。建议列为 V2 方向。

#### 12. 内容新鲜度维护提醒

建议在 Phase B 中，当 GSC 数据可用时，对 datePublished > 12 月且 GSC 下降的页面生成"建议更新"提醒。与 K09 内容衰退合并实现。
