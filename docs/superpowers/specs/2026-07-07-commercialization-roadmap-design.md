# Veris 商业化路线图 + UI「傻瓜式」体验改版 · 总体任务计划

> 日期：2026-07-07。本文档是**总体路线图（master roadmap）**：定义阶段、子项目、验收标准与依赖顺序。
> 每个子项目（SP）在开始执行时，按仓库惯例先出各自的逐任务 TDD 实现计划（`docs/superpowers/plans/`），再动代码。
> 背景审查结论：平台端到端闭环已真实跑通（628 测试全绿），当前瓶颈是「交付外壳」而非诊断能力。

## 已拍板的决策（2026-07-07 用户确认）

| 决策点 | 结论 |
|---|---|
| 视觉方向 | **浅色专业风 + 动效点缀**——保留现有「炼图术 Studio」`--ds-*` token 体系与 Apple 风浅色系，不换色板 |
| 动效浓度 | **克制点缀**——只在高价值时刻用动效；禁用 WebGL 背景/粒子类 |
| 计划范围 | **全量**——G1 种子交付 + G2 UI 改版 + G3 商业验证 + G4 V1 商业化 |

## 阶段总览与依赖

```
G1 种子交付硬缺口 (≈2 周) ──┐
G2 UI/交互改版 (≈1.5 周，可与 G1 并行 60%) ──┤→ 部署 → G3 种子用户商业验证 (4-6 周，回测周期)
                                              │
                     G3 出现付费信号后才启动 → G4 V1 商业化 (≈4-6 周)
```

**铁律**：G3 验证期内不写 G4 代码。「2-3 个用户愿意付费或拿真实项目试点」（plan-ux §12）是 G4 的启动闸门。

# Phase G1 · 种子交付硬缺口（6 个子项目）

## SP-G1a 生产部署上线

**目标**：拿到一个可以发给种子用户的 URL。
**范围**：
- `vercel.json`（region、函数超时配置）+ Vercel 项目接入
- Turso 生产库 + `drizzle-kit push` 生产迁移流程
- Inngest 生产环境（`INNGEST_EVENT_KEY` / `INNGEST_SIGNING_KEY`）接线验证
- `docs/runbooks/deploy.md`：环境变量清单（从 `.env.example` 整理）、部署步骤、回滚方式
**验收**：生产 URL 上零 key 最小闭环（爬取→findings→建议→prompt）完整跑通一次；Inngest 云端函数触发成功。
**依赖**：无。**建议最先做**——后续所有验证都在真环境做。

## SP-G1b 多项目 + 项目/run 历史列表

**目标**：消除单项目模型，SEO 顾问可管多个客户站并找回历史报告。
**范围**：
- 新路由 `app/[locale]/projects/page.tsx`（项目列表：域名、最近 run 状态、下次回测时间）
- `app/[locale]/projects/[id]/page.tsx` 或项目卡展开：该项目 run 历史（时间、run_type、findings 数、状态）
- 首页 `/` 改为：有项目→项目列表；无项目→新建向导（与 SP-G2a 衔接）
- 清除 `getPrimaryProject()` 的「取第一个」假设，所有引用点改为显式 projectId 传参
- `project_settings`（GSC token、siteUrl）确认按 project 隔离（schema 已支持，主要是读取路径改造）
**验收**：创建 2 个项目各跑 1 次诊断，列表页能区分并回访各自 run；settings 的 GSC 授权按项目独立。
**依赖**：无，可与 SP-G1a 并行。

## SP-G1c 设置页 API Key 录入 UI（BYOK 落地）

**目标**：非技术用户不碰 env 文件即可配齐数据源。
**范围**：
- 新表 `provider_credentials`（project 级或全局级，值 **AES-256-GCM 加密存储**，主密钥走 `CREDENTIALS_ENCRYPTION_KEY` env）
- 设置页数据源矩阵升级：每行「未配置」→ 内联录入框 + 「测试连接」按钮（真打一次最小 API 调用返回 ✓/✗ 及错误原因）
- 读取优先级：DB credentials > env 变量（保持现有 env 方式向后兼容）
- `buildProbeProvidersFromEnv()` 等工厂改为 `buildProbeProviders(credentials)`
**验收**：全程只在 UI 里录入 OpenAI key → 探针数据源变 configured → 跑 run 出 AI 可见度真数据；DB 里查到的是密文。
**依赖**：无。**注意**：SP-G1f 的 GSC token 加密复用同一加密模块，先做本项。

## SP-G1d 屏 1 GSC 开关接真 OAuth

**目标**：修复首次体验最大断裂——新建表单勾了 GSC 却什么都不发生。
**范围**：
- 新建流程中「连接 GSC」变为真实状态：未授权→点击发起 `/api/gsc/auth`（`state` 带 returnTo 回到向导当前步骤）；已授权→显示 ✓ 与站点选择
- callback 后回流到发起页而非固定 `/settings`
- 未连接继续前进时，显式提示「关键词与排名数据将缺失，诊断置信度降低」（对齐 plan-ux §9.2 空态原则）
**验收**：从新建向导内一路完成 GSC 授权并回到向导原位置；跳过授权时提示语出现且 run 正常降级。
**依赖**：与 SP-G2a（向导化改版）同一片代码，**建议合并进 SP-G2a 的计划一起执行**。

## SP-G1e 报告 PDF 导出 + 只读分享链接

**目标**：报告能正式交付给客户（交付最后一公里）。
**范围**：
- 分享链接：`report_shares` 表（run_id + 随机 token + 可选过期时间）→ 公开只读路由 `app/share/[token]/page.tsx`（无 Shell 导航、无操作按钮、带「由 Veris 生成」页脚）
- PDF：优先方案 = 复用已接入的 Cloudflare Browser Rendering 对分享页打 PDF（`/pdf` endpoint）；未配 Cloudflare 时降级为引导浏览器打印（现有 PrintButton）+ 打印样式打磨（`@media print`）
- report 页工具栏加「生成分享链接 / 导出 PDF」
**验收**：无登录态浏览器打开分享链接可看完整报告；PDF 分页正常、中文字体不缺字。
**依赖**：SP-G1a（Cloudflare PDF 需要生产 URL 可达）。

## SP-G1f 方法论与安全补漏

**目标**：护住「每个结论可验证」的招牌 + 消除明文凭据。
**范围**：
- 把已有的 `assertFindingClaimEvidence`（`lib/repositories/validators.ts`，有单测）接入 `createFindings` 写路径：`measured_hard` 无 L4 证据 / `measured_sample` 无 L3-L4 证据时写入即抛错——这是 plan-ux §6.2 唯一未落地的约束
- `project_settings.gsc_refresh_token` 改加密存储（复用 SP-G1c 加密模块），含存量数据迁移
**验收**：构造违规 finding 写入被拒的回归测试；DB 中 refresh_token 为密文且 GSC 采集仍正常。
**依赖**：SP-G1c（加密模块）。工作量最小（≈半天–1 天），**可作为 G1 第一个热身任务先做 L4 约束部分**。

# Phase G2 · UI/交互「傻瓜式」改版

## 设计原则（约束所有 G2 子项目）

1. **可信感 > 炫酷感**。证据严肃性是产品护城河，动效永远不得出现在证据标签、claim_type 呈现、数字本体的语义层上。
2. **每屏一个主动作**。用户任何时刻都能回答「我现在该点哪」。
3. **引导而非配置**。缺数据源不是报错，是一条带按钮的路：「缺什么 → 为什么需要 → 一键去修」。
4. **动效 = 状态变化的解释器**。只用于：进度推进、数字首次呈现、抽屉/卡片展开、完成时刻。装饰性动效一律不加。
5. **浅色为基**。沿用 `--ds-*` token；改版只允许新增 token，不允许改既有色值语义。

## React Bits 使用规范（调研结论落地）

**接入方式**：shadcn CLI 拷贝 **TS + Tailwind 变体**源码进 `components/fx/` 目录，我们拥有并维护这些源码。License：MIT + Commons Clause（可商用，不可把组件库本身当商品卖）——对 Veris 无影响。

**允许清单（CSS/轻量级，叶子级 `'use client'`）**：

| 组件/模式 | 用在哪 | 价值 |
|---|---|---|
| CountUp（数字滚动） | StatStrip 指标卡首次进入视口 | 「测出来了」的仪式感 |
| BlurText / SplitText（一次性进场） | 报告页大标题、诊断完成标题 | 交付时刻的品质感 |
| SpotlightCard（hover 聚光） | 建议卡 RecCard、项目列表卡 | 引导视线到当前操作对象 |
| ShinyText（流光） | 仅限「诊断完成」「回测可用」等一次性状态徽章 | 提示新状态；**禁止**用于「实测」等证据标签 |
| 进度条 shimmer / 阶段步进动画 | RunProgress 采集阶段 | 等待期的确定感 |
| AnimatedList（逐条滑入） | 证据流事件、findings 首次渲染 | 「证据正在累积」的叙事 |

**禁用清单**：一切 WebGL/three.js 背景（Aurora/Silk/Particles/StarField）、Dock、磁吸、页面级转场、循环播放的装饰动画。新增依赖上限：`motion/react` 一个；**禁止引入 GSAP / three.js / @react-three/fiber**。
**无障碍**：所有动效组件必须尊重 `prefers-reduced-motion`（降级为直接呈现终态）。

## SP-G2a 新建分析向导化（含 SP-G1d GSC 接线）

**目标**：把单页大表单变成 3 步向导，首次使用零门槛。
**范围**：
- `NewAnalysisForm` 重构为向导：
  - **第 1 步「你的网站」**：只填域名 + 行业/市场（默认值智能预填：输入域名后自动探测语言/市场给默认）
  - **第 2 步「连接数据」**：GSC 授权卡（SP-G1d 的内嵌 OAuth）+ AI 探针 key 状态卡（未配→链接到设置页锚点或内联录入，复用 SP-G1c）；每张卡三态：已连接 ✓ / 未连接（附影响说明）/ 跳过
  - **第 3 步「确认与预估」**：诊断范围摘要 + 预计耗时 + 预计 API 成本（plan-ux §9.1 屏 1 要求，现缺失）+ 大按钮「开始诊断」
- 竞品输入从必填文本降级为「可选，AI 会自动发现候选，稍后确认」（competitors 页已有候选确认流）
**验收**：新用户从落地到发起第一次 run ≤ 2 分钟且无需看文档；每步只有一个主 CTA；GSC 授权在向导内闭环。
**依赖**：SP-G1c、SP-G1d。

## SP-G2b 数据源健康度常驻 + 空态 CTA 化

**目标**：把「pending 一片空白」变成「一条修复之路」。
**范围**：
- Shell 顶栏加数据源健康度指示（如 `数据源 3/6`），点击展开抽屉：每个数据源状态 + 缺失影响 + 「去连接」按钮（直达设置页对应锚点）
- 现有 pending 块（StatStrip/PresenceMap/SovBar 已有 configHint）统一升级为标准空态组件 `EmptyStateCTA`：图标 + 「缺少 X 数据源」+ 影响一句话 + 主按钮
- 首个 run 完成后若有数据源缺失，报告顶部显示「本次诊断基于 N/6 数据源，补齐后可提升覆盖」横幅
**验收**：环境只配一半 key 时，每个空白区域都有明确出路，无死胡同空态。
**依赖**：SP-G1c（「去连接」的目的地）。

## SP-G2c 诊断进行中体验（RunProgress 叙事化）

**目标**：3-10 分钟的采集等待从「转圈」变成「看得见的工作」。
**范围**：
- RunProgress（已有 SSE 事件流）升级为阶段故事线：当前阶段大字号 + 已完成阶段打勾折叠；证据流事件用 AnimatedList 逐条滑入（「已抓取 37 页」「Perplexity 探针 12/20」）
- 关键计数用 CountUp
- 完成时刻：标题 BlurText 进场 +「查看诊断结果」主按钮
- 失败态：哪个阶段失败、错误原因、可重试按钮（现有 failed 标记的 UI 化）
**验收**：全程可旁观每个采集阶段推进；拔掉某个 key 触发 provider error 时失败原因可读且可重试。
**依赖**：无（可最早启动的 G2 项）。

## SP-G2d 报告页品质打磨

**目标**：报告 = 产品的脸面 = 传播素材。
**范围**：
- 排版升级：目录侧栏锚点导航、章节间距节奏（用 `--sp-section`）、打印样式完整过一遍
- 首屏「诊断概要卡」：总分/五支柱雷达或条形 + 证据等级图例（向客户解释 L0-L4 的可信度阶梯——这是差异化卖点，要显性化）
- 数字首现 CountUp、大标题 BlurText（仅首次）
- 与 SP-G1e 分享页共用同一套渲染，保证分享出去的就是打磨后的版本
**验收**：分享链接打开的报告在手机上可读；打印/PDF 无断版；证据等级图例出现在概要区。
**依赖**：SP-G1e。

**G2 执行顺序建议**：G2c（无依赖）→ G2b → G2a → G2d。

# Phase G3 · 商业验证（非代码，与 G1/G2 上线后并行）

| # | 动作 | 产出/判据 |
|---|---|---|
| G3-1 | 找 3-5 个真实外贸独立站（自有资源/社群招募），免费完整诊断 | 每站一份分享链接报告 |
| G3-2 | 陪跑执行：用户按建议改页面/发内容，记录采纳率 | 采纳了哪些建议、拒绝原因 |
| G3-3 | 4-6 周后同协议回测，产出 delta 报告 | ≥2 个站有可展示的 before/after |
| G3-4 | 内容启动：2-3 篇「诊断实录」（脱敏证据抽屉截图 + 回测 delta）发外贸社群（米课圈/知无不言）+ 公众号/知乎 | 每篇带预约诊断入口 |
| G3-5 | Dogfooding 公开数据：用 Veris 定期监测若干公开外贸站在各 AI 引擎的可见度，做成行业观察帖 | 月度一期，积累信任资产 |
| G3-6 | 付费意愿访谈：报完 delta 后问「续跑回测 + 月度监控，¥XXX/月做不做」 | **≥2 个愿付费 → 启动 G4；否则回炉定位** |

# Phase G4 · V1 商业化（启动闸门：G3-6 通过）

> 以下为范围级定义；每项启动前单独出 TDD plan。技术选型标注了推荐项，执行时再最终确认。

| SP | 内容 | 关键点 |
|---|---|---|
| G4a 账号与多租户 | 登录 + org/user 模型，全表加 org 归属与行级隔离 | 推荐 Auth.js v5（Drizzle adapter，与 libSQL 兼容）；求快可用 Clerk |
| G4b 订阅计费 | Stripe Checkout + webhook + 套餐额度（项目数/prompt 数/回测频率），对齐 plan-ux §10.3 定价带 | 国内收款另评估（虎皮椒/Stripe 人民币） |
| G4c 周期性自动重跑 | Inngest cron 按套餐排期回测，完成后邮件通知 + 报告链接 | 复用现有 retest 链，主要是调度与通知 |
| G4d 白标报告 | Agency 套餐：分享页/PDF 换客户 logo、隐藏 Veris 品牌 | 调研确认这是代理商采购第一决策因素 |
| G4e Prompt set 管理 | 行业模板库 + 自定义 prompt set（同协议回测约束：set 版本化，跨版本禁止直接对比） | 长期护城河容器 |
| G4f 成本控制 | provider 调用计量、缓存、单 run 成本上限熔断、用量仪表 | BYOK 之外托管 key 套餐的前提 |
| G4g 审计与团队 | 操作日志（谁接受了哪条建议）、多成员角色 | Agency 场景需要 |
