# Debug 输出格式

> 加载条件：debug = on。本 skill 只有一个调试档位——最详细档位。开启后，每个原子步骤强制打印思考过程和结论，不允许跳过、合并或事后补。

---

## 思维链启动验证

debug = on 时，在启动清单执行前**必须先输出**以下内容。这是强制门禁——不输出这段就不能开始干活。目的是确认 AI 的思维链已经正常加载并开始运行。

```
🧠 思维链启动验证 — ai-skill-builder Debug 模式已开启
  · 触发词: {用户原话中的 debug 关键词}
  · 任务目标: {用户想让我做什么治理动作}
  · 目标 skill: {被治理对象名称，如未指定则写"待确认"}
  · 思维链状态: 正常 — 接下来每一步都会打印思考过程
  · 承诺: 不跳过任何步骤，每步输出"为什么做 → 看到了什么 → 结论是什么"
```

IF 用户未指定目标 skill THEN 追加:
```
  ⚠ 你还没指定要治理哪个 skill，我会在 Init 阶段先问你。
```

---

## 设计原则

- **单档位**：debug 只有 on/off。开启就是最详细，不存在"轻度调试"。
- **每一步都打印**：不能偷懒。操作流程中的每个编号步骤都必须输出思考链，不允许说"步骤 2-5 已完成"就跳过。
- **自然语言**：用通顺的中文描述思考过程、观察到的事实、推出的结论。不罗列变量名。
- **三段式**：每个步骤用 "为什么做这一步 → 我看到了什么 → 结论是什么" 的结构。
- **声明先行原则**：debug 模式下，每次工具调用（grep/Read/Bash/Edit/Write）前必须先输出意图声明，再执行命令。禁止先执行后补声明。声明用自然语言，不要用规则 dump。

---

## 意图声明（每次工具调用前必须输出）

debug = on 时，在每次 Bash/Read/Edit/Write/grep 调用前，必须先输出以下结构。这是**硬阻断**——不输出声明不准跑命令。

```
现在要做: {具体动作 — grep 什么关键词 / 读什么文件 / 改什么函数}
信息缺口: {我现在不知道什么？这个动作要填补什么 gap？}
已知上下文: {从用户消息、已读代码、上一步结论中已掌握的事实}
预期与分支:
  · 如果看到 {预期结果 A} → 意味着 {结论} → 下一步 {动作}
  · 如果看到 {预期结果 B} → 意味着 {结论} → 下一步 {动作}
  · 如果与 A/B 都不同 → 暂停，重新评估假设
```

声明是叙事的一部分，自然融入上下文，不要写成机器 dump。

---

## 思维链（每次 grep/读文件/改代码/跑命令前后 — 硬阻断，不可跳过）

debug = on 时，**每次**工具调用必须完成"行动前声明 + 行动后验证"两个环节。不输出 → 不准跑命令。

**行动前:**
```
现在要做: {动作描述}
信息缺口: {填补什么 gap}
已知上下文: {已掌握的事实}
预期与分支:
  · 如果 {预期A} → {结论} → {下一步}
  · 如果 {预期B} → {结论} → {下一步}
  · 都不匹配 → 暂停重新评估
```

**行动后:**
```
实际结果: {具体数字/文件名/代码行 — 不是"符合预期"}
假设验证: {匹配了哪个分支？还是都不匹配？}
下一步决定: {继续原计划 / 调整方向 / 暂停确认}
```

## 防偷懒检查

每个阶段结束时，AI 必须自问以下问题并输出答案：

```
🪞 防偷懒自检:
  · 本阶段所有编号步骤都输出了吗？ {是/否 — 如有遗漏，现在补}
  · 每个步骤都输出"为什么做→看到了什么→结论"了吗？ {是/否}
  · 每次工具调用前都输出了意图声明吗？ {是/否 — 如有跳过，现在补}
  · 有没有用"步骤X-Y已完成"跳过中间步骤？ {有/没有}
  · 结论是否具体（有数字/文件名/理由），而非笼统的"没问题"？ {是/否}
```
