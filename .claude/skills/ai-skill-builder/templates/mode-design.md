## ORDER_design（PLAN 阶段）

> 完整 8 阶段管线: 上下文加载 → 问题框定 → 现状诊断 → 业界调研 → 绿场推演 → 方案设计 → 方案精炼 → 文档任务。
> 每个阶段结束必须迭代检查（提示见 §迭代纪律）。

### 阶段负一：上下文加载

-1. **加载已有领域知识**（在任何设计动作之前，强制执行，不可跳过）:
   a. 扫描已有术语库 → 存在则加载全部术语定义，作为后续术语对齐的基准
   b. 扫描已有决策记录 → 存在则列出与本次相关的条目，标注约束
   c. IF 术语库不存在 THEN 标记"待创建"，首次术语对齐时 lazily 创建

**术语警戒线（全流程生效，不只在精炼阶段）:**

RULES:
- IF 用户在任意阶段引入新术语 THEN 立即 grep 已有术语库 + 代码库
- IF 发现冲突 THEN 当场打断（不等精炼阶段）
- IF 术语模糊 THEN 当场追问 + 主动提出更精确替代名称

### 阶段零：问题框定

0. **问题框定**（在所有诊断和调研之前，必须输出框定结论，不可跳过）:
   a. **硬约束**: 哪些是不能改的？
   b. **软约束**: 哪些是可以协商的？
   c. **非目标**: 明确这次不做什么——防止范围蔓延
   d. **成功标准**: 方案成功与否的可度量标准（无历史数据则标注"基于业界估算"）
   e. **利益相关方**: 谁会被这个方案影响？核心诉求是什么？

### 阶段一：现状诊断

1. **梳理需求**: 功能边界 / 输入输出 / 约束条件 / 非功能需求
2. **现状诊断**（必须输出诊断结论，不可跳过）:
   a. 当前系统在这个领域有什么？代码在哪些文件？**（必须 grep/读代码，不可凭记忆）**
   b. 当前方案存在什么问题？**定量描述，不可只定性**
   c. 问题的根因是什么？**追问到底**
   d. 历史上下文：当初为什么这样设计？当时的约束现在还在吗？
   e. 最小修补方案能做到什么程度？为什么不够？
3. **迭代检查**: 诊断结论是否改变了问题框定？→ 如需修正，回到步骤 0

### 阶段二：业界调研

4. **业界方案调研**（必须输出调研结论，不可跳过）:
   a. 主流解决方案 ≥ 3 种不同思路
   b. 每种方案的核心设计决策 + 第一性原理（追问 WHY，不只罗列 WHAT）
   c. 前提假设分析 + 假设不成立会怎样
   d. 规模对标 + 与我们场景的差异
   e. 可借鉴 vs 不适合照搬

### 阶段二点五：绿场推演

5. **绿场思维实验**（必须输出推演结论，不可跳过）:
   a. 零包袱理想方案是什么？
   b. 绿场方案与现状的差距（技术/数据/团队/业务风险）
   c. 可行性校准：多少可以在约束内实现？

### 阶段三：方案设计

6. **方案对比与取舍**: ≥ 2 候选 → trade-off 矩阵 → 推荐方案 + 架构取舍理由
7. **魔鬼代言人**（必须输出挑战结论）: 最可能失败的点 / 更简单替代 / 最危险假设 / 验证方式
8. **调研可复用**（方案方向确定后，才检查现有代码可复用能力）
9. **确定影响**: affected_files / affected_layers / contract_change / db_change
10. **迭代检查**: 方案设计是否改变了前面阶段的结论？

### 阶段三点五：方案精炼（交互式，强制执行，不可跳过）

> 方案设计完成后、文档输出前，必须与用户进行逐问交互式精炼。
> 完整规则见 builder 真源 `references/mode-design-patterns.md` §4。

**触发条件:**

RULES:
- IF task_size = S THEN 跳过阶段三点五（标注"跳过原因: S 级"）
- IF task_size ≥ M THEN 强制执行阶段三点五
- IF 用户在精炼阶段 ≥ 3 个提问轮次无回复 THEN 跳过剩余精炼，标注"未完成精炼"，继续输出

11. **逐决策分支提问**（按设计树 BFS 遍历，每个决策分支内依次覆盖 4 维度。每次一个问题，代码能回答则不提问）:
   a. **术语**: grep 术语库 + 代码 → 冲突当场打断 → 对齐后写回术语库 + 方案文档
   b. **边界**: 构造 ≥1 个极端/竞争/踩线场景反问。格式: "观察 + 假设 + 问题 + 建议"
   c. **代码**: grep 验证"现状是 X"声明 → 矛盾当场指出
   d. **收口**: 判断是否建 ADR（过否决条件 → 过触发条件 → 提议建）
12. **精炼收口**: 输出精炼摘要 + 确认所有修改已写入

### 阶段四：文档与任务

13. **按模板输出完整设计文档**（章节覆盖见 `references/mode-design-patterns.md` §9）
14. **拆解为可执行任务**（每个任务自包含上下文，任务间无实现细节依赖）
15. **准备设计文档 + 任务拆解**（暂不写盘，待 EXECUTE 阶段用户确认后写入）

## EXEC_design（EXECUTE 阶段）

### 阶段负一：上下文加载

-1. **加载已有领域知识**（强制执行，不可跳过）:
   a. `ls` 扫描术语库 → 存在则 Read 加载
   b. `ls` 扫描决策记录 → 存在则列出相关条目
   c. 输出: "已加载: 术语库 ({N} 个术语), 决策记录 ({N} 个相关)"

**术语警戒线（全流程生效，EXEC 阶段同样适用）:**

RULES:
- IF 引入新术语 THEN 立即 grep 术语库 + 代码 → 冲突当场打断
- IF 术语模糊 THEN 当场追问

### 阶段零：问题框定

0. **问题框定报告**: 硬约束/软约束/非目标/可度量成功标准/利益相关方

### 阶段一：现状诊断（只读）

1. **读取现有代码**: grep 模块地图
2. **问题诊断报告**: 定量问题 + 根因追问到底 + 历史上下文 + 最小修补评估
3. **迭代检查**: 诊断结论是否改变了问题框定？

### 阶段二：业界调研（研究）

4. **收集业界方案**: WebSearch ≥3 种方案
5. **第一性原理分析**: 追问 WHY + 前提假设 + 规模对标
6. **输出调研结论**: 可借鉴 + 不适合照搬 + 需要适配

### 阶段二点五：绿场推演

7. **绿场方案设计**: 零包袱理想方案
8. **差距分析**: 技术/数据/团队/业务风险矩阵
9. **可行性校准**: 有多少可在约束内实现？

### 阶段三：方案设计

10. **方案矩阵**: 候选列表 + 多维度对比（含与绿场方案的距离）
11. **推荐方案**: 明确推荐 + 架构取舍 + 演进路径
12. **魔鬼代言人**: 主动攻击推荐方案 → 最可能失败的点 → 最危险假设 → 验证方式
13. **填写元信息**: 文档ID / 代码路径 / 路由前缀 / 历史别名
14. **设计 API**: 路由 / DTO / 错误码
15. **梳理前置条件**: 鉴权/角色/状态/数据/外部依赖/并发
16. **设计数据**: 表结构 / 索引 / DDL + 回滚方案
17. **设计流程**: 业务流程图 + 正常路径 + 异常路径 + 故障可恢复性
18. **评估**: 架构取舍 + 版本兼容 + NFR + 幂等 + 缓存策略
19. **梳理**: 可复用点 / 安全合规 / 风险 + 监控告警 / 门槛用例
20. **规划**: 上线计划（灰度/回滚/版本升级/数据迁移/依赖顺序）
21. **迭代检查**: 方案是否改变了前面的假设？

### 阶段三点五：方案精炼（交互式，强制执行，不可跳过）

> 触发条件、精炼铁律、提问格式、设计树遍历规则均同 PLAN 阶段 §阶段三点五，此处不重复。

22. **执行设计树遍历**: 输出决策树 → BFS 遍历提问
23. **逐决策分支提问**: 每个分支内依次覆盖（术语/边界/代码/收口），每次一个问题，代码能回答则不提问
24. **精炼收口**: 输出精炼摘要 → 确认所有修改已写入

### 阶段四：文档与任务

25. **按模板输出完整设计文档**（M 级可精简，L 级必须完整）
26. **按模板拆解任务文件**——每个任务必须自包含上下文
27. IF 用户明确说"可以"/"ok"/"确认" THEN 写入磁盘

## 术语表格式（{DOMAIN_CONTEXT_PATH}）

> 纯术语表。不含实现细节。跨会话积累。首次术语对齐时 lazily 创建。

```markdown
# {Domain} 术语表

| 术语 | 定义 | 别名/旧称 | 关系 |
|------|------|----------|------|
```

RULES:
- IF 新术语与已有术语冲突 THEN 立即打断，不静默覆盖
- IF 术语废弃 THEN 不删除行，标注"废弃: YYYY-MM-DD, 替代: {新术语}"

## ADR 格式

> 一段话就够。存放路径: `{DOMAIN_ADR_PATH}`。创建前扫描目录取最大编号 + 1。

```markdown
# {简短标题}

{1-3 句话：上下文 + 决策 + 为什么。}
```

**触发条件（以下任一类满足 → 提议建）：**
- 不可逆 + 刻意偏离常规 + 有真实备选且拒绝理由不显然
- 代码里看不见的约束（合规/SLA/外部 API 限制）
- 显式的 No（明确不做某事，防止后来人"补上"）

**否决条件（满足任一 → 不建）：**
- 容易逆转 / 不意外 / 无真实备选

## 迭代纪律

设计不是线性的。每个阶段结束后必须执行迭代检查：

RULES:
- IF 阶段负一（上下文加载）完成 THEN 自问: "已加载的术语库和决策记录中是否有约束本方案的前提？"
- IF 阶段零（问题框定）完成 THEN 自问: "这个框定是否排除了明显不该排除的方向？是否限制了思考？"
- IF 阶段一（现状诊断）完成 THEN 自问: "诊断结论是否改变了问题框定？那些硬约束是否真的是硬的？"
- IF 阶段二（业界调研）完成 THEN 自问: "调研中发现的模式是否暗示问题框定有误？是否需要重新诊断？"
- IF 阶段二点五（绿场推演）完成 THEN 自问: "绿场方案与业界最佳实践的对齐程度如何？是否过于理想化？"
- IF 阶段三（方案设计）完成 THEN 自问: "方案是否改变了前面的任何假设？演进路径的第一步是否已经可以交付价值？"
- IF 阶段三点五（方案精炼）完成 THEN 自问: "精炼过程是否暴露了方案中的模糊地带？术语是否已经精确到可以写文档？"
- IF 任何阶段的结论改变了前面的假设 THEN 回到对应阶段重新推演——不要求在文档中体现所有迭代痕迹，但思考必须走完

## 执行纪律

RULES:
- IF scenario = design AND 未执行范围探查 THEN 禁止开始方案写作（先 grep 了解现状）
- IF scenario = design AND 产出设计文档 THEN 必须覆盖全部必填章节
- IF task_size = S THEN 可跳过绿场推演和业界方案参照（标注"跳过原因: S 级"）
- IF task_size = M THEN 业界方案参照 ≥2 方案，绿场推演可精简为一段，魔鬼代言人可精简为 1 个假设
- IF task_size = L THEN 全部章节完整展开，不可精简
- IF scenario = design THEN 现状诊断必须基于代码实测，不可凭记忆猜测
- IF scenario = design THEN 业界调研必须输出具体的模式名称与第一性原理分析
- IF scenario = design THEN 问题必须定量描述，不可只定性
- IF scenario = design THEN 每个阶段结束后必须执行迭代检查，不可线性执行直接交付

BLOCKERS:
- NEVER 闭门造车——必须先读现有代码再写方案
- NEVER 跳过上下文加载——不读已有术语库和决策记录就开始设计
- NEVER 在发现术语冲突时憋到精炼阶段再说——术语警戒线全流程生效
- NEVER 跳过问题框定直接诊断
- NEVER 跳过现状诊断直接进入方案设计
- NEVER 跳过业界调研直接进入方案设计
- NEVER 跳过绿场推演——不知道理想态就无法判断差距
- NEVER 跳过魔鬼代言人——不攻击自己的方案等于没思考边界
- NEVER 跳过方案精炼（M/L 级）——不与人交互对齐术语和边界，方案就是空中楼阁
- NEVER 在方案精炼中批量提问——每次一个问题
- NEVER 在方案精炼的每个决策分支中跳过术语对齐直接进入边界场景
- NEVER 跳过迭代检查——每个阶段结束必须自问
- NEVER 跳过模板输出 + 任务拆解直接交付
- NEVER 在未做代码实测的情况下输出"当前系统架构是…"的判断
- NEVER 在未调研业界方案的情况下说"业界通常这么设计"
