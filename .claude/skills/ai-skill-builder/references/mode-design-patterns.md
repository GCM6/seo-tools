# Mode 设计规范

> 通用规则集。适用于任意 skill 的 `modes/*.md` 文件设计——不限领域。
> 提炼自 `ai-backend` design mode 的多轮重构。
> 本文件是 skill builder 在"新建/审查 mode 文件"时的强制参考真源。

---

## 0. Mode 哲学声明（可选）

复杂 Mode 文件（>200 行、>5 阶段）应在开头包含"哲学声明"区块，用 RULES/BLOCKERS 桶承载核心理念：

```markdown
## 设计哲学

{一句话：这个 Mode 的核心理念是什么？}

```
{ASCII 流程图：管线全貌}
```

RULES:
- IF {场景} THEN {核心原则 1}
- IF {场景} THEN {核心原则 2}
- IF {场景} THEN 新发现可能修正旧结论——允许回到上一阶段重新推演

BLOCKERS:
- NEVER {绝对禁令 1}
- NEVER 线性执行后直接交付——每个阶段结束后必须迭代检查
```

RULES:
- IF Mode 行数 > 200 THEN 强烈建议加哲学声明
- IF Mode 行数 ≤ 200 THEN 哲学声明可选
- IF 哲学声明 THEN 流程图 + RULES + BLOCKERS 三者缺一不可
- IF 哲学声明中的 BLOCKERS THEN 与执行纪律 BLOCKERS 保持一致但粒度不同（哲学=概要禁令，纪律=操作禁令）

## 1. 阶段管线

多阶段 Mode 必须使用编号管线，每个阶段结束处必须有迭代检查点：

```
阶段负一（上下文加载）→ 阶段零（框定）→ 阶段一（诊断）→ ...
  ↓ 每个阶段末尾: 迭代检查（"这个结论是否改变了前面的假设？"）
```

RULES:
- IF Mode 涉及 > 3 个阶段 THEN 使用阶段管线（不是平铺步骤列表）
- IF 阶段管线 THEN 每个阶段末尾必须有迭代检查：自问 "这个结论是否改变了前面阶段的假设？"
- IF 迭代检查发现假设变更 THEN 回到对应阶段重新推演
- IF 阶段编号 THEN 整负数用于"前置准备"（如 -1 上下文加载），零用于"框定"，正数用于执行阶段

BLOCKERS:
- NEVER 线性执行后直接交付——每个阶段结束后必须执行迭代检查

**迭代检查 prompt 模板（按阶段定制，不可用空洞的"自问是否改变了假设"替代）:**

```
- IF 阶段负一（上下文加载）完成 THEN 自问: "已加载的术语库和决策记录中是否有约束本方案的前提？是否有术语已被定义？"
- IF 阶段零（问题框定）完成 THEN 自问: "这个框定是否排除了明显不该排除的方向？是否限制了思考？"
- IF 阶段一（现状诊断）完成 THEN 自问: "诊断结论是否改变了问题框定？那些硬约束是否真的是硬的？"
- IF 阶段二（业界调研）完成 THEN 自问: "调研中发现的模式是否暗示问题框定有误？是否需要重新诊断？"
- IF 阶段二点五（绿场推演）完成 THEN 自问: "绿场方案与业界最佳实践的对齐程度如何？是否过于理想化而忽略了某些现实约束？"
- IF 阶段三（方案设计）完成 THEN 自问: "方案是否改变了前面的任何假设？演进路径的第一步是否已经可以交付价值？"
- IF 阶段三点五（方案精炼）完成 THEN 自问: "精炼过程是否暴露了方案中的模糊地带？术语是否已经精确到可以写文档？"
- IF 任何阶段的结论改变了前面的假设 THEN 回到对应阶段重新推演
```

## 2. PLAN/EXEC 去重

Mode 文件同时有 PLAN 和 EXEC 两套步骤时，EXEC 应引用 PLAN 而非逐字重复：

RULES:
- IF PLAN 和 EXEC 存在相同规则块 THEN EXEC 中写 "规则同 PLAN 阶段 §{章节名}，此处不重复"
- IF EXEC 引用 PLAN THEN 保留执行步骤编号，删除重复的规则阐述
- IF PLAN 步骤数 > 15 OR EXEC 步骤数 > 30 THEN 审查是否有重复块

---

## 3. S/M/L 分层

根据 task_size 分层控制深度，避免轻量任务被过度流程化：

RULES:
- IF task_size = S THEN 可跳过深度分析阶段（标注 "跳过原因: S 级"）
- IF task_size = M THEN 强制核心阶段，允许边缘阶段精简
- IF task_size = L THEN 全部阶段完整展开，不可精简
- IF 分层规则 THEN 在 Mode 文件的"触发条件"和"执行纪律"两处保持一致

## 4. 交互式精炼（Grill 模式）

当 Mode 涉及方案设计或需要人机对齐时，使用交互式精炼模式：

### 4.1 精炼铁律

RULES:
- IF 问题可以通过读代码/grep/读已有文档回答 THEN **不提问**，直接给出发现并一句确认
- IF 必须提问 THEN 每个问题必须附带**推荐答案**（格式: "建议: {推荐选项}，因为 {一句话理由}"）
- IF 提问 THEN 每次一个问题，等待用户反馈后再继续下一个。禁止批量提问
- IF 用户回答模糊（"差不多""应该可以"）THEN 追问到精确为止，不放过模糊答案

### 4.2 提问格式（强制）

```
观察: {从代码/grep/文档中看到的具体事实，含文件:行号}
问题: {一句话}
建议: {推荐选项}，因为 {一句话理由}
```

### 4.3 设计树遍历

ORDER:
1. 列出方案中的所有关键决策点（至少包含: 数据模型选型 / 状态流转策略 / 分层归属 / 外部依赖边界）
2. 标注决策间的依赖关系: "决策 B 依赖决策 A 的结论（A 改变 → B 的选项空间改变）"
3. 输出决策树（缩进表示依赖层级）:
   ```
   根决策: {最早需要做出的、影响面最广的决策}
     ├── 一级分支: {依赖根决策的子决策}
     │     ├── 二级分支: {依赖一级分支的子决策}
     │     └── 二级分支: {...}
     └── 一级分支: {...}
   ```
4. 后续提问顺序: **BFS 遍历决策树**——先解决根决策，再展开每个一级分支，最后处理叶子。术语对齐和边界场景不再作为独立话题，而是穿插在每个决策分支的提问中
5. IF 用户在回答某个决策时改变了前提假设 THEN 回到决策树，修剪受影响的分支，重新确定提问顺序

### 4.4 每个决策分支内依次覆盖 4 维度

| 维度 | 做什么 | 写回 |
|------|--------|------|
| 术语 | grep 已有术语库 → 冲突当场打断。无冲突则确认。术语模糊则主动提出更精确的替代名称 | 写回术语表 + 方案文档 §术语表 |
| 边界 | 构造至少 1 个极端/竞争/踩线场景反问。格式: "观察 + 假设 + 问题 + 建议" | 写回方案文档 §业务规则 |
| 代码 | grep 验证"现状是 X"的声明 → 矛盾当场指出: "你的方案假设 {声明}，但 {file}:{line} 实际行为是 {代码片段}。以哪个为准？" | 写回方案文档 §现状诊断 |
| 收口 | 判断是否建 ADR: 先过否决条件（容易逆转/不意外/无备选 → 不建），再过触发条件（不可逆+刻意偏离+有备选 / 隐式约束 / 显式No → 提议建） | 写回 ADR 或方案文档 §架构取舍 |

### 4.5 超时兜底

RULES:
- IF 用户在精炼阶段 ≥ 3 个提问轮次无回复 THEN 跳过剩余精炼步骤，在产出文档中标注"未完成精炼: 用户未回复"，继续输出结果。提醒用户后续可随时补充精炼

### 4.6 Implement 场景适配（M/L 实现前澄清门禁）

上述精炼模式不仅适用于 Design 场景。当 skill 采用 S/M/L 分级体系时，Implement 场景的 M/L 级任务也应在 PLAN 完成后、EXECUTE 开始前执行澄清门禁——在写代码之前对模糊描述和不确定术语进行强制澄清，避免"AI 自以为理解了但实际理解歪了"。

**与 Design 精炼的区别:**

| 维度 | Design 方案精炼 | Implement 澄清门禁 |
|------|:---:|:---:|
| 时机 | 方案设计完成后、文档输出前 | PLAN 完成后、EXECUTE 开始前 |
| 目标 | 方案正确性 | 实现正确性 |
| 决策来源 | 方案设计中的设计决策 | PLAN 执行步骤中的实现选择 |
| S 级 | 跳过 | 跳过 |

**M 级精简版:**

RULES:
- IF task_size = M THEN 从 PLAN 中提取 ≤3 个关键决策点（关键决策点 = 如果理解错了会导致返工的实现选择）
- IF 决策点 ≤3 个 THEN 全部覆盖；IF >3 个 THEN 只取影响面最大的 3 个，其余标注"未精炼: 超出 M 级配额"
- 每决策点覆盖 2 维度（每次一个问题，精炼铁律同 §4.1）:
  a. **术语**: 主动建议精确命名。格式: "建议字段/常量/状态值命名为 {推荐命名}，因为 {理由}。对吗？"（不开放提问——AI 必须先给出推荐答案）。grep 已有术语库 + 代码检查冲突
  b. **边界**: 空值/零值/超长/重复提交/并发/权限不足时的预期行为是什么？

**L 级完整版:**

RULES:
- IF task_size = L THEN 构建决策树 → BFS 遍历（同 §4.3），每决策覆盖 4 维度（同 §4.4）: 术语 + 边界 + 代码 + 收口
- IF 用户答案改变了 PLAN 中的前提假设 THEN 修剪决策树受影响分支，调整后续提问顺序
- 超时兜底同 §4.5

**触发规则:**

RULES:
- IF scenario = implement AND task_size = M THEN 执行 M 级精简版
- IF scenario = implement AND task_size = L THEN 执行 L 级完整版
- IF task_size = S THEN 跳过实现前澄清

BLOCKERS:
- NEVER 在 M 级用完整版协议
- NEVER 在 L 级用精简版协议
- NEVER 在 M/L implement 任务中跳过澄清门禁直接进入 EXECUTE
- NEVER 批量提问 — 每次一个问题
- NEVER 跳过术语直接问边界 — 模糊的术语会让边界场景测试失去意义
- NEVER 在澄清未完成时开始写代码

## 5. 术语警戒线

术语对齐不只在精炼阶段，应全流程生效：

RULES:
- IF 用户或方案在任意阶段引入了一个新业务术语 THEN 立即 grep 已有术语库 + 代码库中是否已有相同/冲突用语
- IF 发现冲突 THEN 当场打断（不等精炼阶段），格式: "观察: 术语库定义 '{已有术语}' = {定义}。你刚说的 '{新术语}' 指代 {含义}。它们是同一个概念吗？建议: {推荐}"
- IF 术语库尚未创建 AND 用户引入了 ≥ 2 个新术语 THEN 提醒: "建议等术语确定后写回术语库，当前暂记在会话中"
- IF 术语模糊（一个词指代多个概念）THEN 当场追问 + 主动提出更精确的替代名称

BLOCKERS:
- NEVER 在发现术语冲突时憋到精炼阶段再说——术语警戒线全流程生效，发现冲突当场打断

## 6. 上下文加载

Mode 启动时先加载已有知识，再开始设计：

RULES:
- IF Mode 涉及"设计/改造/扩展"语义 THEN 在阶段零之前加阶段负一: 扫描已有术语库 + 已有决策记录
- IF 已有术语库存在 THEN 加载全部术语作为后续术语对齐的基准
- IF 已有决策记录存在 THEN 列出与本次相关的条目，标注"受 XXX 约束，本次方案不能违反此前提"
- IF 术语库不存在 THEN 标记"待创建"，首次术语对齐时 lazily 创建
- IF 相关决策记录未找到但其决策域重叠 THEN 标记"可能的 ADR 候选"，在精炼阶段决策收口时提示

## 7. 术语表格式（CONTEXT.md）

> 纯术语表。不含实现细节、不含架构决策、不含代码路径。跨会话积累。
> 存放路径: `docs/{domain}/CONTEXT.md`
> 创建时机: 第一个术语对齐结论产生时 lazily 创建。

```markdown
# {Domain} 术语表

> 本文件是纯术语表，不含实现细节。架构决策见 docs/design/adr/。

| 术语 | 定义 | 别名/旧称 | 关系 |
|------|------|----------|------|
| {Term} | {一句话精确定义} | {历史名称, 如无则 —} | {is-a / has-a / 等同于 / —} |
```

RULES:
- IF 新术语与已有术语冲突 THEN 立即打断用户对齐，不静默覆盖
- IF 术语定义变更 THEN 更新定义 + 标注变更日期
- IF 术语废弃 THEN 不删除行，标注 "废弃: YYYY-MM-DD, 替代: {新术语}"
- IF 多域项目 THEN 每个域一个 CONTEXT.md，维护索引文件指向各域 CONTEXT.md 路径

## 8. ADR 格式

> 仅当决策满足触发条件时创建。一段话就够。ADR 的价值在于记录"做了决策以及为什么"——不在于填满章节。
> 存放路径: `docs/design/adr/{序号}-{slug}.md`
> 创建前先扫描目标目录取最大编号 + 1。目录不存在则 lazily 创建，编号 0001 起。

**核心格式（强制）：**

```markdown
# {简短标题}

{1-3 句话：上下文是什么，我们决定了什么，为什么。}

就是这样。一个 ADR 可以只是一段话。
```

**可选追加（只在真正需要时才加）：**

```markdown
---
状态: proposed | accepted | deprecated | superseded by ADR-{NNNN}
---

## 备选方案

| 方案 | 优点 | 缺点 | 为什么没选 |
|------|------|------|-----------|

## 后果

{选择这个方案后，什么变容易了？什么变难了？}
```

**触发条件（以下任一类满足 → 提议建 ADR）：**
- 不可逆 + 刻意偏离常规路径 + 有真实备选且拒绝理由不显然
- 代码里看不见的约束（合规要求 / 性能 SLA / 外部 API 限制 / 合作伙伴合同）
- 显式的 No（明确不做某事的决策，防止后来人 "补上"）

**否决条件（满足任一 → 不建 ADR）：**
- 容易逆转 → 直接改就行，不需要记录
- 不意外 → 没人会困惑"为什么这么做"
- 无真实备选 → 只是做了明显的事，没有值得记录的取舍

## 9. 设计文档必备章节

涉及 "方案设计/架构决策" 语义的 Mode，其产出文档必须覆盖：

| 章节 | 说明 | S 级 | M 级 | L 级 |
|------|------|:--:|:--:|:--:|
| 元信息 | 文档ID / 代码路径 / 路由前缀 / 历史别名 | ✅ | ✅ | ✅ |
| 问题框定 | 硬约束 / 软约束 / 非目标 / 成功标准（可度量）/ 利益相关方 | ✅ | ✅ | ✅ |
| 现状诊断 | 定量问题 + 根因追问到底 + 历史上下文 + 不做大改动的后果 | ✅ | ✅ | ✅ |
| 业界方案参照 | ≥3 种不同思路 + 第一性原理 + 前提假设 + 规模对标 | — | ≥2 方案 | ✅ |
| 绿场推演 | 零包袱理想方案 + 差距矩阵（技术/数据/团队/业务风险）+ 可行性校准 | — | 精简为一段 | ✅ |
| 候选方案对比 | 多维度 trade-off 矩阵（含与绿场方案的距离） | — | ✅ | ✅ |
| 魔鬼代言人 | 最可能失败的点 + 更简单替代 + 最危险假设 + 验证方式 | — | 精简为 1 个假设 | ✅ |
| 方案精炼摘要 | 术语对齐 N / 边界场景 N / 代码矛盾 N / ADR N / CONTEXT.md N | — | ✅ | ✅ |
| 演进路径 | 分步交付 + 每步可交付价值 + 每步风险 + 回滚方式 | — | ✅ | ✅ |

---

**章节模板骨架（按需复制到设计文档中）:**

### 问题框定

```markdown
**硬约束（不可协商）:**
| 约束 | 原因 | 影响 |
|------|------|------|

**软约束（可协商）:**
| 约束 | 协商空间 | 影响 |
|------|---------|------|

**非目标（明确不做）:**
| 非目标 | 原因 |
|--------|------|

**成功标准（可度量）:**
| 标准 | 度量方式 | 目标值 |
|------|---------|--------|

**利益相关方:**
| 角色 | 核心诉求 |
|------|---------|
```

### 现状诊断

```markdown
| 问题 | 定量现象 | 根因（追问到底） | 历史上下文 | 影响范围 | 不做大改动的后果 |
|------|---------|----------------|-----------|---------|----------------|
```

### 业界方案参照

```markdown
| 方案/模式 | 核心设计 | 第一性原理 | 前提假设 | 假设不成立会怎样 | 规模对标 | 差异 | 可借鉴点 | 不适合原因 |
|----------|---------|-----------|---------|----------------|---------|------|---------|-----------|
```

### 绿场推演

```markdown
- **理想方案**: {零包袱最优解}
- **与现状的差距**: 技术 / 数据迁移 / 团队 / 业务风险
- **可实现部分**: {约束内可实现的部分}
- **替代方案**: {不可实现部分的替代}
```

### 候选方案对比

```markdown
| 维度 | 方案A: {名称} | 方案B: {名称} | 方案C: {名称} |
|------|:-----------:|:-----------:|:-----------:|
| 核心思路 | | | |
| 与绿场方案的距离 | 近/中/远 | 近/中/远 | 近/中/远 |
| 复用现有能力 | 高/中/低 | 高/中/低 | 高/中/低 |
| 实现复杂度 | 高/中/低 | 高/中/低 | 高/中/低 |
| 扩展性 | 高/中/低 | 高/中/低 | 高/中/低 |
| 与现有架构匹配度 | 高/中/低 | 高/中/低 | 高/中/低 |
| 团队熟悉度 | 高/中/低 | 高/中/低 | 高/中/低 |
| **推荐** | | | **← 推荐方案** |

**推荐理由**: {为什么选这个？最大的权衡是什么？}
```

## 10. 先外后内原则

RULES:
- IF Mode 涉及"方案设计/技术选型/架构决策" THEN 先向外看（业界怎么做），再向内看（有什么可复用）
- IF 调研可复用 THEN 在方案方向确定后执行，不在需求阶段执行——不在错误方向上复用
- IF 业界调研 THEN 追问 WHY，不只罗列 WHAT:
  - 这个方案要解决的根本问题是什么？
  - 它的前提假设是什么？假设不成立会怎样？
  - 它在什么规模下有效？什么规模下失效？
  - 它的维护成本和隐形成本是什么？
