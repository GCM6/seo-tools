# 触发矩阵：症状/诉求 → 主线站点

> 诊断与优化的路由真源。把用户的「一句话症状」映射到主线站点 + 该加载的 reference。
> 用法：先在「症状关键词」里命中，按「先查」顺序加载 reference，再按「常见根因」给归因，最后开方。

## A. 诊断类（scenario = diagnose）

| 用户症状（关键词） | 先查（主线站） | 常见根因 | reference |
| :--- | :--- | :--- | :--- |
| 没流量 / 没排名 / 出词少 | S1 → S2 → S3 | 上游需求不成立、无关键词调研堆产品、无分层架构、技术抓取问题 | 01 → 02 → 03 |
| 不收录 / 新页不收录 | S7 → S3 | 抓取预算被无效页/小语种耗尽、sitemap/robots、新页质量低 | 07 → 03 |
| 有流量但没询盘 | S1 → S4 | 目标词无搜索量、落地页只讲公司不讲场景/价值、承接力差 | 01 → 04 |
| 掉权 / 流量暴跌 / 收录骤减 / 排名集体消失 | S8 | 明显降权（算法/人工处罚/小语种泛滥/内容农场） | 08 |
| 排名停滞 / 卡第二页 / 3-4个月不动 | S7 → S8 | 关键词蚕食、孤岛页无内链、内容时效过期、隐性降权 | 07 → 08 |
| 排名波动 / 算法更新后下跌 | S7 → S6 | 真波动 vs 第三方工具滞后；先以 GSC 区分 | 07 → 06 |
| 核心词被博客文章顶替 | S7 | 关键词蚕食，需内容合并 + 301/canonical | 07 |
| 多个站互相打架 | S2 | 站间权威稀释，需主体隔离 + 关键词错位 | 02（多站点小节） |

## C. 指导类（scenario = new_build）

| 站点阶段 | 切入点 | 顺序 |
| :--- | :--- | :--- |
| none（未建站/规划） | S1 | S1→S2→S3→S4→S5→S6（S7/S8 标为后续） |
| new（<3月） | S5/S6 | 先补 S5 冷启动+外链、S6 监测；回看 S2/S4 是否达标 |
| growth（3-6月） | S6/S7 | 看见效数据、开始 S7 维护 |
| mature（6月+） | S7 | 存量维护、做减法、防过度优化 |
| penalized | S8 | 直接进降权修复 |

## D. 学习类（scenario = learn）

先 `00-glossary.md` 命中术语 → 再加载该术语所属主线 reference（见 `reference-index.md`）。
